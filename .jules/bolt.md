## 2024-05-15 - In-Memory Promise Cache Concurrency

**Learning:** When implementing an in-memory Promise cache (like `cachedPromise` for `getAccessToken`), setting `tokenExpirationTime = 0` during refresh acts as a pending state. If the cache hit logic only checks `Date.now() < tokenExpirationTime`, it will return false for `0`, causing concurrent requests to incorrectly bypass the cache and fetch anew while a fetch is already in progress. Furthermore, transient network or parsing errors can permanently poison the cache with a rejected Promise if `.catch()` does not explicitly reset the cache variable. Also, failed HTTP responses (like 429/500) will corrupt the cache if parsed without checking `!response.ok`.
**Action:** When refreshing an expired cached Promise, explicitly reset the tracking variable to a pending state (e.g., `tokenExpirationTime = 0`) before initiating the new fetch. The cache hit logic must explicitly check for this pending state (e.g., `tokenExpirationTime === 0 || now < tokenExpirationTime`). When fetching, check `!response.ok` before calling `.json()`, and attach a `.catch()` block to the cached promise chain to reset the cache variable (`cachedPromise = null`) before re-throwing the error.
